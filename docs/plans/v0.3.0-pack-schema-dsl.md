## 背景

Pack 架构的核心承诺是：**所有 Pack 通过 manifest YAML 声明 UI，不写
React 代码**。现实是：美正 Pack 落地时为了赶交付，引入了 5 个
Pack-specific `.tsx` 模板：

| 文件 | 行数 | 用途 |
| --- | --- | --- |
| `MeizhengConfig.tsx` | 408 | 配置页根组件，组合 3 个子 editor |
| `MeizhengDashboard.tsx` | 327 | 仪表盘（fuel rate + USD rate + workflow status） |
| `CarrierConfigEditor.tsx` | 380 | 承运商配置（services 映射 + cron + service mapping） |
| `ExchangeRateConfigEditor.tsx` | 309 | 汇率配置（source / conversion / target + 多 schedule） |
| `ZoneConfigEditor.tsx` | 286 | 分区配置（per-carrier zone schedules） |

**这 5 个文件违反架构铁律**。v0.3.0 的目标是：
1. 设计能表达上述 5 个 UI 的 schema DSL
2. 实现通用 schema-driven renderer
3. 将 5 份 manifest 改为 YAML 声明
4. **删除 5 个 `.tsx` 文件**

## 当前 PackConfig schema 的能力（2026-05-17）

参见 `src-tauri/src/ipc/pack/config.rs::PackConfigSchema`：

```rust
pub struct PackConfigSchema {
    pub key: String,
    pub label: String,
    pub field_type: String,    // text/number/secret/select
    pub required: bool,
    pub secret: bool,
    pub description: String,
    pub help: String,
    pub group: String,
    pub validation: String,
    pub placeholder: String,
    pub default: serde_json::Value,
    pub options: Vec<String>,  // for select
}
```

**只能表达扁平的 key-value 字段**。无嵌套 / 无数组 / 无自定义 widget /
无联动 / 无计算字段。

## 美正 Pack 用到但 PackConfig 不支持的能力

| 能力 | 在哪用 | DSL 表达建议 |
| --- | --- | --- |
| 嵌套对象（`source.url` / `conversion.divideBy` / `target.currencyCode`） | ExchangeRate | `nested:` 字段类型 + 子 schema |
| 动态对象数组（schedules: [{ name, cron, ... }]） | ExchangeRate / Zone | `array:` 字段类型 + item schema + min/max |
| 自定义 widget（CronPicker） | 所有 3 个 Editor | `widget:` 字段引用 widget 注册表 |
| 计算/预览字段（`682.78 ÷ ${divideBy} = ...`） | ExchangeRate | `computed:` 模板字符串 |
| 跨字段联动（`enabled` 字段控制其他字段是否渲染） | Carrier / Zone | `showIf:` 表达式 |
| 占位符 + helper 文本 + 验证 | 全部 | 已有 `placeholder` / `help`，加 `pattern` / `min` / `max` |
| 单选 / 多选 / 标签输入 | Carrier (services) | `selectMulti:` / `tag:` 字段类型 |
| 复合 KV 映射（service → carrier service 映射） | Carrier | `keyValueArray:` 字段类型 |
| Workflow 触发按钮 | Dashboard | `actions:` 已有，但需 inline 嵌入到表单 |
| 仪表盘卡片（fuel rate 显示 + 状态指示） | Dashboard | 用现有 MetricsCard / DataTable 组合 |
| 实时数据刷新（`packViewData` IPC 拉数据） | Dashboard | `dataSource:` 引用 IPC + 轮询配置 |

## DSL 草案

```yaml
# meizheng/views/exchange-rate-config.view.yaml
template: SchemaConfig
title: 汇率自动更新配置
description: 配置中行美元汇率的抓取计划
config_key: exchange-rate-config        # 哪个 ~/.hermes/pack-data/<id>/<key>.yaml
schema:
  - key: source
    label: 数据源
    type: nested
    fields:
      - key: url
        label: 数据源 URL
        type: text
        placeholder: https://www.boc.cn/sourcedb/whpj/
      - key: rateType
        label: 汇率类型
        type: text
        placeholder: 现汇卖出价
      - key: queryKeyword
        label: 搜索关键词
        type: text
        placeholder: 美元
        help: 数据源页面上的货币名称
      - key: earliestTime
        label: 抓取时间起点
        type: time
        help: 取此时间后第一笔汇率
  - key: conversion.divideBy
    label: 除以系数
    type: number
    width: small
    preview: "示例: 682.78 ÷ {value} = {{value | divide:682.78}}"
  - key: schedules
    label: 抓取计划
    type: array
    addLabel: 添加计划
    minItems: 1
    item:
      - key: name
        label: 计划名称
        type: text
        placeholder: 计划名称（如 早盘抓取）
      - key: cron
        label: 调度
        type: cron        # 引用 widget 注册表中的 CronPicker
        showPreview: true
  - key: target.currencyCode
    label: 货币代码
    type: text
    width: half
  - key: remarkTemplate
    label: 备注模板
    type: text
    width: half
    placeholder: "更新汇率{datetime}  {rate}"
```

## 实现路径

### Phase 3a · 基础设施（2-3 天）

**1. 后端**
- 扩展 `PackConfigSchema` 为递归 enum：`Field::Scalar` / `Field::Nested` / `Field::Array` / `Field::Computed`
- 兼容旧 schema：旧的扁平 schema 自动归一为 `Field::Scalar` 列表
- IPC contract 不变（YAML serialize 出来就是新形式）

**2. 前端 widget 注册表**
- `src/features/pack/templates/SchemaConfig/widgets/registry.ts`
- 内置：`text` / `number` / `secret` / `select` / `time` / `cron` / `tag`
- 注册时机：模块加载时静态注册

**3. 通用 renderer**
- `src/features/pack/templates/SchemaConfig.tsx`：递归遍历 schema，dispatch 到 widget
- 支持 `showIf` 表达式（用 jsep 或简版自定义 parser）
- 支持 `preview` 模板字符串（mustache 风格）
- 支持 `array` 的 add/remove/update（用 `shared/useArrayField` hook）

### Phase 3b · 美正 Pack 迁移（2-3 天）

**优先级（从简到繁）**：
1. `MeizhengDashboard` → `CompositeDashboard` + `MetricsCard` + `WorkflowLauncher` 组合（已有模板，几乎不改 schema renderer）
2. `ExchangeRateConfigEditor` → `SchemaConfig`（中等：嵌套 + 动态数组）
3. `ZoneConfigEditor` → `SchemaConfig`（同上）
4. `CarrierConfigEditor` → `SchemaConfig`（最复杂：多碳运商 list + service 映射）
5. `MeizhengConfig` → `CompositeDashboard`（包含上面 3 个 SchemaConfig view）

每迁完一个：
- 删原 .tsx
- 跑 vitest + Playwright（如有美正 specs）
- 用户人工 smoke 验证 UX 还原

### Phase 3c · 其他 Pack 受益（按需）

未来跨境电商 Pack 直接用 SchemaConfig，**无需写任何 React**。

## 实际落地与本草案的差异（2026-05-17 收尾）

Phase 3a + 3b 当日全部完成（8 个 commit，净 −1474 行）。最终 DSL 比草案多出 3 项能力，权威 schema 见 `@/Users/zbin/AI项目/CoreyOS/src-tauri/src/pack/manifest.rs::ConfigField` 与 `@/Users/zbin/AI项目/CoreyOS/src/features/pack/templates/SchemaConfig.tsx`：

### 1. `record` 字段类型（草案未规划）

`Record<string, NestedShape>` 字典型字段。UI 表现：点 "+" → `window.prompt` 输入 key → 渲染为可折叠卡片，每张卡片是 key 标题 + `fields:` 子表单。等价于老 `CarrierConfigEditor` / `ZoneConfigEditor` 的"承运商动态字典"。

```yaml
- key: carriers
  type: record
  label: 承运商
  key_label: 承运商 ID（如 ups / usps / fedex）   # prompt 文案，默认 "键"
  add_label: 添加承运商
  min_items: 1                                     # 与 array 同义
  fields:                                          # 每个 value 的子 schema
    - key: enabled
      type: bool
    - key: services
      type: array
      item: [...]
```

### 2. `default:` 深合并（草案未规划）

任意 record / nested / 标量字段都可以声明 `default:`。加载时执行 `mergeDefaultsIntoData(schema, diskData)`：

- **record**：磁盘缺的 key 用默认补，磁盘有的 key 保留（再向下递归合并）
- **nested**：递归进子 schema
- **标量**：仅当 `cur === undefined | null | ''` 时用默认

替代了老 `buildDefaultZoneConfig()` 那种 "in-memory 注入默认 + 用户没保存就丢" 的反模式 —— 现在默认是声明式的，下次保存会自动持久化到磁盘。

实例：`@/Users/zbin/AI项目/CoreyOS/src-tauri/assets/skill-packs/meizheng/manifest.yaml` 的 zone-config 视图 `carriers.default` 写了 ups/usps/fedex 三组默认值。

### 3. `pack_named_config_*` 通用 IPC（草案叫 `config_key`）

视图通过 `config_file: <slug>` 声明读写 `~/.hermes/pack-data/<id>/config/<slug>.yaml`。`slug` 必须 `[a-z0-9][a-z0-9-]*`，由 Rust 端 `validate_config_name` 锁死防路径穿越。

**额外组合**：视图 `schema:` 内联但 `config_file:` 不写 → 走老 `pack_config_get/set`（带 snake_case ↔ camelCase transform 链路）。系统配置视图靠这条复用了 fuel-rate-config.yaml，避免数据分叉。

### 4. 草案里"草拟但未实现"的部分

- **`{{value | divide:682.78}}` mustache filter**：没做。`fillTemplate` 只支持 `{path}` 直接插值。要派生值就用 `type: computed` + `preview:` 文本拼接，或在数据源 JS 里预算好。
- **`schema_version` 字段**：没加。manifest 直接用最新 schema，老 Pack 的扁平 schema 通过 `#[serde(default)]` 兼容（所有新字段都 default 空值）。
- **`template: <CustomComponentName>` escape hatch**：保留，但目前 0 个 Pack 用。所有美正视图 100% manifest 化。

## 风险

1. **showIf 表达式语法选择**：`jsep`（依赖 +1MB）vs 自实现简化 parser（更轻但不灵活）。倾向自实现：只支持 `eq` / `ne` / `and` / `or`，足够 99% 用例。
2. **DSL 兼容老 PackConfig**：迁移过程中两种 schema 并存，需要 schema_version 字段标记。
3. **UX 像素级回归**：现有 `.tsx` 用 hand-rolled Tailwind class，schema-driven 用 widget 默认 style。预期 5-10% 像素差，需要 widget 支持 `className` override 兜底。
4. **DSL 表达力不够时的回退**：保留"escape hatch"机制，manifest 可以声明 `template: <CustomComponentName>` 调用基座注册的 React 组件——但**只允许基座注册**，Pack 不能注入。这条铁律不破。

## 与 v0.3.0 的关系

跨境电商 Pack 需要的视图（战场地图 / AI 总管 / 广告机器人 / 库存机器人 /
差评机器人 / 数据分析 / 市场分析 / 战场雷达 / 六维诊断）大部分是数据驱动
仪表盘和表单。**SchemaConfig DSL 是 v0.3.0 的前置依赖**——v0.3.0 启动前
必须先把这个 DSL 落定，否则跨境电商 Pack 也会重蹈美正覆辙。

## 不在本计划内（明确拒绝的方向）

1. **JSON Schema / OpenAPI 全套** ——过度工程，CoreyOS 自己的轻 DSL 足够
2. **运行时 schema 编辑器（GUI）** ——Pack 作者用 YAML 即可
3. **跨平台 schema 验证**（如 Pydantic-style）——前端用 TypeScript type guard 即可
4. **SchemaConfig 嵌入 chat 工具调用结果** ——超出 v0.3.0 scope
