## 三、功能设计

### 3.1 Sprint 1 — 费用估算 + 时间筛选（P0）

#### 3.1.1 费用估算

**数据源**：现有 `analytics_summary` 已返回 `prompt_tokens` + `completion_tokens`，加上 `MODEL_PRICES` 价格表。

**新增字段**（`AnalyticsTotals`）：
```rust
estimated_cost_usd: f64,    // 总估算费用（USD）
estimated_cost_cny: f64,    // 总估算费用（CNY，汇率 7.2）
```

**计算逻辑**（Rust 端）：
- 按 session 的 `model` 字段匹配 `MODEL_PRICES` 前缀
- `cost = (prompt_tokens / 1M × input_price) + (completion_tokens / 1M × output_price)`
- 没有 model 信息的用 `FALLBACK_PRICE`

**新增 API**（`analytics_cost_breakdown`）：
```rust
struct CostBreakdown {
    total_usd: f64,
    total_cny: f64,
    by_model: Vec<ModelCost>,      // 每个模型的费用
    daily_cost: Vec<DayCost>,       // 30 天每日费用
}

struct ModelCost {
    model: String,
    prompt_tokens: i64,
    completion_tokens: i64,
    cost_usd: f64,
}

struct DayCost {
    date: String,
    cost_usd: f64,
}
```

**UI**：
- KPI 卡片新增"费用"卡片（`$12.34 / ¥88.85`），金色高亮
- 新增"30 天费用趋势"折线图（在 token 趋势下方）
- "Top 模型"条形图改为双轴：左侧 count，右侧费用标签

#### 3.1.2 时间筛选器

**选项**：`7天` / `30天` / `90天` / `全部`

**实现**：
- 前端 `useState<DateRange>('30d')` 控制
- IPC 传参 `days: Option<i64>`，Rust 端 `WHERE created_at >= (now - days * 86400000)`
- `全部` 传 `None`，不加时间过滤
- 默认 30 天（兼容现有行为）

**影响**：
- `analytics_summary` IPC 增加可选 `days` 参数
- 前端 URL 不变，纯 state 切换

### 3.2 Sprint 2 — 延迟追踪 + Token 拆分（P0）

#### 3.2.1 延迟追踪

**数据采集**：
- 在 `chat.rs` 的流式响应中，记录两个时间戳：
  - `first_token_at`: 第一个 content delta 到达时间（TTFT）
  - `last_token_at`: 最后一个 delta 到达时间
- 存入 `messages` 表新增列：
  ```sql
  ALTER TABLE messages ADD COLUMN first_token_latency_ms INTEGER;
  ALTER TABLE messages ADD COLUMN total_latency_ms INTEGER;
  ```

**DB migration**（v7）：
- 新增两列，默认 NULL（历史数据无延迟）

**聚合**（`analytics_latency`）：
```rust
struct LatencyStats {
    p50_ms: i64,
    p95_ms: i64,
    p99_ms: i64,
    avg_ms: i64,
    by_model: Vec<ModelLatency>,
}
```

**UI**：
- KPI 新增"平均延迟"卡片（`1.2s`）
- 新增"延迟分布"图（P50/P95/P99 三条线，30 天趋势）

#### 3.2.2 Token 拆分展示

**数据已有**：`prompt_tokens` 和 `completion_tokens` 已经在 `AnalyticsTotals` 里。

**UI 修改**：
- KPI "总 token"卡片改为双行展示：`输入: 12,345 / 输出: 6,789`
- 30 天 token 趋势图改为堆叠面积图（下方 prompt，上方 completion）
- "Top 模型"条形图每行增加 token 拆分标签

### 3.3 Sprint 3 — 错误率 + Channel 拆分 + 导出（P1）

#### 3.3.1 错误率

**数据源**：`messages` 表已有 `error` 字段（非 NULL = 出错）。

**新增聚合**：
```rust
struct ErrorStats {
    total_errors: i64,
    error_rate: f64,              // errors / total_messages
    daily_errors: Vec<DayCount>,
    top_error_types: Vec<NamedCount>,  // 按错误消息前 50 字符归类
}
```

**UI**：
- KPI 新增"错误率"卡片（`2.3%`，红色警告 >5%）
- 30 天趋势图增加错误数虚线叠加层

#### 3.3.2 Channel 拆分

**数据源**：`sessions` 表已有 `adapter_id`，但 channel 信息在 gateway session 同步时未持久化。

**需要**：
- `sessions` 表新增 `source_channel TEXT` 列（`wechat` / `telegram` / `cli` 等）
- gateway session 同步时填充此字段

**聚合**：
```rust
channel_usage: Vec<NamedCount>,  // 按来源平台
```

**UI**：
- 新增"按来源"水平条形图（微信 / 钉钉 / TG / CLI / Web）

#### 3.3.3 CSV 导出

**实现**：
- Analytics 页右上角增加"导出 CSV"按钮
- 前端拼 CSV（从已有 analytics 数据）
- 使用 `@tauri-apps/plugin-dialog` 的 `save()` 选保存路径
- `@tauri-apps/plugin-fs` 写文件

**导出内容**：
- 摘要 CSV：日期、消息数、token 数、估算费用、错误数
- 详细 CSV（可选）：每条 message 的 session_id、model、tokens、latency、费用

### 3.4 Sprint 4 — 预算联动 + 高级图表（P2）

#### 3.4.1 预算联动

- Analytics 页底部显示 Budget 进度条（复用 Budgets 页组件）
- 费用趋势图上叠加预算上限虚线

#### 3.4.2 高级图表

- "模型对比"雷达图：费用 vs 延迟 vs 质量（反馈率）
- "使用热力图"：7×24 热力图（哪天哪个时段用得多）

## 七、验收标准

1. Analytics 页顶部显示费用估算（USD + CNY），数值合理
2. 时间筛选器可切换 7/30/90/全部，所有图表联动
3. 延迟 P50/P95/P99 数据准确（与手动计时对比）
4. Token 拆分（输入/输出）在 KPI 和趋势图中可见
5. 错误率 >5% 时显示红色警告
6. CSV 导出文件可在 Excel/WPS 正确打开
7. CI 全绿（macOS + Windows + Ubuntu + E2E）
