# 02 · 产品问题诊断

**时间**: 2026-04-25
**类型**: 分析
**触发**: 用户指令 — "你觉得当前产品有什么问题"

---

## 诊断方法

基于以下信息源进行诊断：
1. 全部源代码 (Rust 后端 + React 前端)
2. 项目文档 (docs/ 目录 20+ 文件)
3. 产品审计报告 (10-product-audit-2026-04-23.md)
4. 积压项 (06-backlog.md)
5. Hermes Reality Check (hermes-reality-check-2026-04-23.md)
6. 路线图 (05-roadmap.md)

---

## 问题清单

### 一、战略层

| # | 问题 | 严重度 | 说明 |
|---|---|---|---|
| S1 | 上游依赖风险 | 高 | Hermes API/配置格式非公开标准，通道 schema 推断错误已导致 3/8 静默失败 |
| S2 | 多 Agent 叙事超前 | 中 | Claude Code / Aider 适配器仍为 Mock，Phase 5 "多 Agent 控制台" 尚未兑现 |
| S3 | 功能复制倾向 | 中 | 调度器、Memory、Skills 都经历了"先自建后发现 Hermes 已有 → 删掉重来" |

### 二、产品设计

| # | 问题 | 严重度 | 说明 |
|---|---|---|---|
| P1 | 侧边栏 18 个入口过于臃肿 | 高 | 用户认知负荷高，低频功能与高频功能并列，违背键盘优先设计 |
| P2 | 首页 Onboarding 缺渐进引导 | 高 | 所有步骤同时展示，"通道"步骤永远无法自动完成，缺少 30 秒快速体验路径 |
| P3 | 差异化功能打磨不足 | 中 | Compare / Trajectory / Budgets 缺乏深度分析洞察和导出能力 |
| P4 | 离线/弱网体验缺失 | 中 | Gateway 离线时无可操作建议，断连后无自动重连，无本地消息队列 |

### 三、工程

| # | 问题 | 严重度 | 说明 |
|---|---|---|---|
| E1 | IPC 类型安全手工维护 | 高 | 80+ IPC 命令的 TS 类型手工镜像 Rust，任何不同步都静默失败 |
| E2 | 缺少 Error Boundary | 中 | 任何功能模块崩溃会导致整个应用白屏 |
| E3 | ChatRoute 复杂度过高 | 中 | 单组件承担会话/消息/附件/预算/搜索/路由等 8+ 职责 |
| E4 | 文档与代码不同步 | 中 | 架构文档中 sqlx、keyring、shiki 等依赖实际不存在 |

### 四、测试与质量

| # | 问题 | 严重度 | 说明 |
|---|---|---|---|
| T1 | E2E 依赖 Mock | 中 | Playwright 在 Mock Tauri 环境下运行，真实集成验证不足 |
| T2 | 性能预算未 CI 自动化 | 低 | 冷启动/RAM 目标手动验证，bundle size 脚本存在但未集成到 gate |

---

## 优先级排序

| 优先级 | 问题 | 建议行动 |
|---|---|---|
| P0 | P1 侧边栏臃肿 | 合并低频页面，精简到 ~10 个入口 |
| P0 | P2 首页引导 | 设计渐进式引导 + 30 秒快速体验 |
| P1 | E4 文档不同步 | 清理 01-architecture.md 虚构依赖 |
| P1 | E3 ChatRoute 复杂度 | 拆分为 3-4 个 Hook + 子组件 |
| P1 | E2 Error Boundary | 为每个功能路由添加错误边界 |
| P2 | E1 IPC 类型安全 | 引入 tauri-specta 自动生成 |
| P2 | P4 离线体验 | 本地消息队列 + 网络状态提示 |
| P2 | T2 性能 CI | 性能预算检查加入 CI pipeline |

---

## 结论

CoreyOS 在架构设计和工程完成度上优秀（4 天完成 8 个 Phase），但存在 **功能堆砌但打磨不足** 的问题。下一步应该是做减法和做深度，而不是继续加功能。
