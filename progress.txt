## 浏览器焦点抢夺 — 调研已完成（不要重做）

### 已实测/已验证的死路

| 方案 | 结果 |
| --- | --- |
| `--headless=new` | UPS Akamai ERR_HTTP2 拦截 |
| Patchright headless（默认配置） | Google 超时、UPS ERR_HTTP2、Amazon 202 bot challenge |
| Patchright README 的 best practice (`channel: "chrome", headless: false`) | 仍需要窗口 = 仍抢焦点 |
| Camoufox | Firefox fork，0% 检测但 headed only，焦点问题不解 |
| AppleScript 移窗口到 Space | macOS 不暴露 Spaces API，需要 Yabai（要部分关 SIP）|
| 改 `/Applications/Google Chrome.app/Contents/Info.plist` 加 LSUIElement | Chrome 是 hardened runtime + library-validation + notarized，改完无法启动 |
| `--window-position=-2400,-2400` 隐藏 | Chrome 启动时从 Local State 恢复保存位置，flag 静默失效 |

### Chromium 10 年根本 bug
- chromium#223828, #40770130, #40182355
- 任何 CDP 命令都触发 macOS NSApplication activate

### 唯一可行路线（用户拍板，工程量 1-2 天）
**自带 Playwright/Patchright 的 Chromium 二进制，改其 Info.plist 加 LSUIElement=YES，ad-hoc 重签**。
- Chromium 不是 hardened runtime/notarized → 可以改可以重签
- LSUIElement=YES → 没 Dock 图标、没菜单栏、CDP 命令不激活窗口
- 真 Chromium 二进制 → TLS 指纹和 Chrome 95% 一致 → Akamai 大概率通过（已验证）
- 跨平台对称：macOS = LSUIElement，Windows = SW_HIDE（现有 `hide_chrome_window` 已实现），Linux = window-position

## Spike 结果（2026-05-11 19:30，PASS）

### 关键纠错

- 不能用 `LSUIElement=true`：Chrome for Testing 设上后 ICU 加载失败（`icudtl.dat not found in bundle` / `Invalid file descriptor to ICU data received`），进程直接退出
- **必须用 `LSBackgroundOnly=true`**：Chromium 启动正常，CDP 工作，且语义更强（永远不会被激活）
- 不能用 `cp -R` 复制 .app：会破坏 bundle 的代码签名 metadata，同样导致 ICU 加载失败
- **必须用 `ditto`** 复制：保留 xattr + 签名元信息
- 不需要重签名：`ditto` + 只改 Info.plist，连 `codesign` 都不用跑，bundle 仍能启动（ad-hoc 签名容忍 Info.plist 修改）

### 验证步骤

ditto "$HOME/Library/Caches/ms-playwright/chromium-1217/chrome-mac-arm64/Google Chrome for Testing.app" /tmp/test-chromium.app
